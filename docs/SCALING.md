# Mythos Scaling-Laws Study

This document describes the scientific contribution of the Mythos project: a small-scale empirical reproduction of the Chinchilla scaling laws (Hoffmann et al., 2022) using the Mythos architecture, run end-to-end on a single MacBook Air M3 (16 GB).

## Goal

Fit the Hoffmann et al. parametric loss surface

```
L(N, D) = E + A · N^(-α) + B · D^(-β)
```

from a sweep of `(N, D)` points generated by training Mythos at 3 model sizes, each to roughly Chinchilla-optimal compute (`D ≈ 20·N`). Compare the fitted exponents `(α, β)` against the Chinchilla reference values (`α ≈ 0.34, β ≈ 0.28`).

A successful reproduction at this scale demonstrates:
1. The implementation has no silent quality bugs (matching exponents implies matching loss-vs-compute curves).
2. The data pipeline is clean enough to recover canonical scaling behavior.
3. The methodology generalizes — the same pipeline could in principle be scaled up to 500M+ given GPU compute.

## Protocol

### Models

| Tag | N (non-embed) | d_model | layers | heads (Q/KV) | d_ff | Config |
|-----|---------------|---------|--------|--------------|------|--------|
| 10m | ~10M  | 256 | 6  | 8/4 | 1024 | `configs/model/10m.yaml` |
| 30m | ~30M  | 384 | 8  | 8/4 | 1536 | `configs/model/30m.yaml` |
| 80m | ~80M  | 512 | 12 | 8/4 | 2048 | `configs/model/80m.yaml` |

All models share: 32K BPE vocab, RoPE θ=10000, RMSNorm pre-norm, SwiGLU, weight tying, GQA with 2:1 Q:KV ratio, `seq_len=1024`.

### Token budgets (Chinchilla-optimal, D ≈ 20·N)

| Tag | Target tokens | Steps × tokens/step |
|-----|---------------|---------------------|
| 10m | 200M | 6,100 × 32,768 |
| 30m | 600M | 24,400 × 24,576 |
| 80m | 1.6B | 48,800 × 32,768 |

`tokens/step = batch_size · grad_accum · seq_len`.

### Training recipe (held constant)

- Optimizer: AdamW (β₁=0.9, β₂=0.95, ε=1e-8), weight decay 0.1 on matmul weights only.
- Schedule: linear warmup → cosine decay to 10% of peak LR.
- Peak LR scales with model size: 6e-4 / 5e-4 / 4e-4 (rough `1/√d_model` heuristic).
- Precision: bfloat16 on MPS.
- Grad clip: 1.0.
- Data: same FineWeb-Edu + code mixture, identical tokenizer across all runs.

### Logging

Every `eval_every` steps we record `(step, tokens_seen, train_loss, val_loss)` to JSONL. The fit script consumes this directly.

## Reproducing

```bash
# 1. Prepare data once
python scripts/prepare_data.py --target medium

# 2. Train each size (sequentially)
python scripts/train.py --config configs/model/10m.yaml --data data/medium
python scripts/train.py --config configs/model/30m.yaml --data data/medium
python scripts/train.py --config configs/model/80m.yaml --data data/medium

# 3. Fit and plot
python scripts/fit_scaling_laws.py \
  --runs checkpoints/scaling_10m checkpoints/scaling_30m checkpoints/scaling_80m \
  --out docs/figures/scaling.png
```

## Expected outcome

With clean data and a correct implementation, the fitted exponents should land within ~20% of the Chinchilla values. Larger deviation usually signals one of:

- LR too high/low for at least one model size (curves don't reach minimum).
- Data leakage between train and val splits (val loss biased low).
- Numerical precision issues (bf16 underflow in attention softmax at the small scale).

The fit script reports per-point residuals, making the failure mode visible.
